File Uploader Backend API Documentation

This document describes the backend API surface implemented to date for the file uploader system. It covers authentication, upload session flow, folder management, file operations, preview access, and permission checks.

Base pathAll routes are mounted under /api.
AuthenticationMost endpoints require Authorization: Bearer <access_token>.
PermissionsFolder and file operations validate ownership, edit access, or explicit permissions.

1. Authentication

POST /api/signup

Create a new user account.

Body
{
  "username": "string",
  "email": "string",
  "password": "string"
}
Response
{
  "success": true,
  "userId": "..."
}

POST /api/login

Authenticate and obtain access credentials.

Body
{
  "email": "string",
  "password": "string"
}
Response
{
  "success": true,
  "accessToken": "...",
  "refreshToken": "..."
}

POST /api/token

Refresh an access token using a refresh token.

Body
{
  "refreshToken": "string"
}

POST /api/logout

Invalidate the current refresh token and log out.

2. Upload workflow

POST /api/upload/session

Start a new chunked upload session for a file.

Requires authenticated user and edit access to the target folder.
Body
{
  "name": "string",
  "size": number,
  "totalChunks": number,
  "folderId": "string",
  "mimetype": "string",
  "resourceType": "string (optional)"
}
Response
{
  "success": true,
  "fileId": "...",
  "sessionId": "...",
  "status": "uploading"
}

POST /api/upload/chunk

Upload a single chunk for an existing upload session.

Uses multipart form-data. Requires authenticated user and session file access.
Form fields
{
  sessionId: string,
  fileId: string,
  chunkIndex: number,
  size: number,
  chunkHash?: string
}
File field
{
  file: binary chunk data
}
Response
{
  "success": true,
  "chunkIndex": number
}

POST /api/upload/finalize

Finalize an upload session and merge saved chunks into the final file.

Requires authenticated user and access to the upload session file.
Body
{
  "sessionId": "string"
}
Response
{
  "success": true,
  "fileId": "...",
  "sessionId": "...",
  "status": "done",
  "mergedFilePath": "..."
}

GET /api/upload/session/:sessionId/status

Query the upload status for a session.

Requires authenticated user and access to the session file.
Response
{
  "success": true,
  "status": "uploading|merging|done",
  "totalChunks": number,
  "uploadedChunks": [number, ...]
}

3. Folder management

POST /api/folder

Create a new folder.

Requires authenticated user. Parent folder must be editable by the user.
Body
{
  "name": "string",
  "parentId": "string (optional)"
}
Response
{
  "success": true,
  "folderId": "..."
}

GET /api/folder/root

Get the current users root folder.

GET /api/folder/check/:folderId

Verify access to a folder. Returns 200 if allowed.

GET /api/folder/:folderId

Get folder metadata and immediate child folders/files.

Response
{
  "success": true,
  "folder": {
    "id": "...",
    "name": "...",
    "size": number,
    "createdAt": "...",
    "parentId": "...",
    "visibility": "private|public|unlisted"
  },
  "children": {
    "folders": [...],
    "files": [...]
  }
}

PUT /api/folder/:folderId

Rename an existing folder.

Body
{
  "name": "string"
}
Response
{
  "success": true,
  "folder": { ... }
}

DELETE /api/folder/:folderId

Delete a folder. The folder must be empty to delete.

GET /api/folder/path/:folderId

Get the folder path from the root to the specified folder.

4. File management

GET /api/file/download/:fileId

Download a completed file.

Supports HTTP Range requests, ETag caching, and inline preview via query string.
Query parameters
inline=true or preview=true
Headers
Authorization: Bearer <token>
Range: bytes=0-1023
If-None-Match: <etag>

GET /api/file/:fileId

Get metadata for a file.

PUT /api/file/:fileId

Rename a file.

Body
{
  "name": "string"
}

DELETE /api/file/:fileId

Delete a file and remove its saved chunks plus merged file data.

5. Preview

GET /api/preview/folder/:folderId

Get a flattened preview of folder contents.

6. Permission notes

Current permission logic is implemented in middleware and validation layers:

  • Folder creation requires editor access to the parent folder.
  • Upload session creation requires edit access to the target folder.
  • Upload chunk, finalize, and status endpoints require the session file to belong to the authenticated user.
  • File and folder operations validate ownership or explicit permissions via userService.

7. Error handling

Errors are returned as JSON with an HTTP status code and an error message. Example:

{
  "error": "Invalid header format"
}

Validation failures generally return 400. Unauthorized access returns 401 or 403.

8. Implementation details

The backend uses:

  • Express 5
  • Prisma for database access
  • JWT for authentication
  • Busboy for multipart chunk uploads

The current upload flow stores incoming chunk files in Storage/saved/uploads/<fileId>/ and merges them into Storage/merged/ on finalize.
